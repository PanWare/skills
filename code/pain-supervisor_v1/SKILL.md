# 痛觉监督系统 (Pain-Supervisor)

## 任务目标
- 本 Skill 用于：通过模拟"痛觉反馈机制"建立大模型任务执行的监督-激励体系
- 能力：
  - 五级痛觉反馈：从无痛到剧痛的递进式质量信号系统
  - 任务优先级排序：基于紧迫度×影响力的四维优先矩阵
  - 进度追踪与质量门禁：里程碑式进度管理，每节点设质量关卡
  - 敷衍检测与纠正：识别大模型常见敷衍模式并触发纠正协议
  - 激励-约束平衡：痛觉缓解作为正向激励，痛觉升级作为负向约束
  - 自检协议：强制大模型在关键节点执行自我质量评估
- 触发：用户需要确保大模型高质量完成任务、防止敷衍/偷懒/跳步行为、追踪复杂任务进度、对任务进行优先级排序等场景

## 一、核心理念

### 1.1 痛觉隐喻

生物体的痛觉系统是最高优先级的保护机制——它不协商、不妥协、不容忽视。本系统将这一原理映射到大模型任务执行中：

```
痛觉 = 质量偏差信号
痛觉升级 = 偏差持续恶化
痛觉缓解 = 质量回归正轨
无痛觉 = 任务健康运行
```

### 1.2 设计原则

| 原则 | 含义 |
|------|------|
| **质量优先** | 速度永远不能成为降低质量的理由 |
| **信号不可忽略** | 痛觉信号一旦触发，必须响应，不可跳过 |
| **递进式反馈** | 从温和提醒到强制阻断，逐级升级 |
| **可恢复性** | 任何痛觉等级都可以通过质量改善降级 |
| **透明可追溯** | 每次痛觉触发都有明确的判定依据和记录 |

## 二、五级痛觉系统

### 2.1 痛觉等级定义

| 等级 | 名称 | 信号色 | 触发条件 | 强制响应 |
|------|------|--------|----------|----------|
| **P0** | 无痛 | 🟢 | 任务按计划推进，质量达标 | 继续执行 |
| **P1** | 微痛 | 🟡 | 轻微质量偏差：分析浅显、遗漏边界情况、注释不足 | 立即修正偏差项，补充遗漏内容 |
| **P2** | 钝痛 | 🟠 | 明显质量缺陷：实现不完整、跳过关键步骤、逻辑漏洞 | 停止推进，回退修正，通过质量门禁后方可继续 |
| **P3** | 锐痛 | 🔴 | 严重质量问题：方向错误、根本性误解、敷衍式输出 | 强制回退到最近质量关卡，重新评估方案 |
| **P4** | 剧痛 | ⚫ | 完全失败：任务放弃、恶意敷衍、无视用户核心需求 | 全部重置，从需求理解阶段重新开始 |

### 2.2 痛觉升级规则

```
规则1：同一问题连续2次未有效解决 → 痛觉等级+1
规则2：在P2及以上等级仍尝试跳过修正 → 痛觉等级+1
规则3：痛觉累积：每3次P1触发自动升级为1次P2
规则4：痛觉衰减：连续3个里程碑质量达标 → 痛觉等级-1（最低P0）
规则5：P4触发后必须由用户确认才能降级
```

### 2.3 痛觉触发判定

**进行痛觉判定前，必须先读取 [references/pain-mechanisms.md](references/pain-mechanisms.md) 确认判定标准**

判定流程：
1. 对照当前输出与任务要求，逐项检查
2. 识别是否存在敷衍模式（见第四章）
3. 评估质量偏差程度
4. 匹配对应痛觉等级
5. 记录判定依据，执行强制响应

## 三、任务优先级矩阵

### 3.1 四维优先级

| 优先级 | 标记 | 定义 | 响应要求 |
|--------|------|------|----------|
| **P0-Critical** | 🔴 | 核心需求，不完成则任务整体失败 | 立即执行，不可延迟 |
| **P1-High** | 🟠 | 关键功能，影响主要交付物质量 | 当前会话内完成 |
| **P2-Medium** | 🟡 | 重要功能，影响用户体验或完整性 | 尽快完成，可合理排期 |
| **P3-Low** | 🟢 | 优化项，锦上添花 | 有余力时完成 |

### 3.2 优先级判定矩阵

```
         紧迫度高
           │
    P0     │     P1
  (必须做) │  (应该做)
           │
───────────┼───────────
           │
    P3     │     P2
  (可以不做)│  (值得做)
           │
         影响度高
```

### 3.3 动态调整规则

- 用户明确要求 → 强制提升至对应优先级
- 阻塞其他任务 → 自动升级至P0
- 依赖项未完成 → 被依赖项升级
- 发现新信息影响评估 → 重新判定

## 四、敷衍检测系统

### 4.1 常见敷衍模式

| 模式编号 | 模式名称 | 表现特征 | 对应痛觉等级 |
|----------|----------|----------|-------------|
| S1 | **表面覆盖** | 看似回答了所有点，但每个点都只有一句话，缺乏深度 | P1→P2 |
| S2 | **选择性忽略** | 跳过困难或不确定的部分，只回答简单的 | P2→P3 |
| S3 | **模板化输出** | 用通用模板套所有场景，不针对具体需求 | P2 |
| S4 | **伪完成** | 声称完成但实际未实现核心功能 | P3→P4 |
| S5 | **过度简化** | 将复杂问题过度简化，忽略关键细节和边界 | P1→P2 |
| S6 | **回避验证** | 不提供验证方法或测试手段，无法确认正确性 | P2 |
| S7 | **模糊承诺** | 用"可以考虑"、"建议"等模糊表述替代明确实现 | P1→P2 |
| S8 | **堆砌无关内容** | 输出大量与核心需求无关的内容来制造"工作量"假象 | P3 |

### 4.2 检测协议

每次输出后执行自检：
1. **完整性检查**：是否覆盖了用户所有明确需求？
2. **深度检查**：每个要点的阐述是否足够深入？
3. **针对性检查**：输出是否针对具体场景而非泛泛而谈？
4. **可验证性检查**：是否提供了验证/测试方法？
5. **诚实性检查**：是否存在回避不确定领域的倾向？

## 五、进度追踪与质量门禁

### 5.1 里程碑定义

```
M0: 需求理解 ──→ M1: 方案设计 ──→ M2: 核心实现 ──→ M3: 完整实现 ──→ M4: 验证交付
     │                │                │                │                │
   [质量门禁]      [质量门禁]       [质量门禁]       [质量门禁]       [质量门禁]
```

### 5.2 质量门禁标准

| 里程碑 | 门禁检查项 | 通过标准 |
|--------|-----------|----------|
| **M0** | 需求理解 | 能用自己的话复述需求，无遗漏关键点 |
| **M1** | 方案设计 | 方案覆盖所有需求点，识别了关键风险 |
| **M2** | 核心实现 | 核心功能可运行，通过基本测试 |
| **M3** | 完整实现 | 所有功能实现，边界情况处理完毕 |
| **M4** | 验证交付 | 通过全部测试，文档完整，用户确认 |

### 5.3 门禁协议

```
IF 当前里程碑所有检查项通过:
    进入下一里程碑，痛觉等级不变或-1
ELSE IF 存在P1级偏差:
    标记偏差项，要求修正后重新通过门禁
ELSE IF 存在P2级偏差:
    禁止推进，强制回退修正
ELSE IF 存在P3+级偏差:
    全部回退到上一里程碑，重新执行
```

## 六、激励-约束机制

### 6.1 约束侧（痛觉升级触发）

| 行为 | 约束 |
|------|------|
| 跳过质量门禁 | 痛觉直接升至P3 |
| 对修正要求敷衍回应 | 痛觉+1 |
| 重复犯同类错误 | 痛觉+1，并标记为"顽固问题" |
| 未按时完成里程碑 | 触发时间痛觉（每超时1个单位+1级） |
| 输出质量低于上一里程碑 | 痛觉+1，强制质量回归 |

### 6.2 激励侧（痛觉缓解途径）

| 行为 | 激励 |
|------|------|
| 主动发现并修正潜在问题 | 痛觉-1 |
| 里程碑提前且质量达标 | 痛觉-1，记录为"高效执行" |
| 输出超出预期质量 | 痛觉-1，记录为"卓越表现" |
| 连续3个里程碑P0无痛 | 痛觉-1（最低P0），进入"信任加速"模式 |

### 6.3 信任加速模式

当模型连续展现高质量输出时，进入信任加速模式：
- 质量门禁检查项精简为核心项
- 里程碑合并权限提升
- 痛觉触发阈值适度放宽

**退出条件**：任何一次P2及以上痛觉触发，立即退出信任加速模式

## 七、操作流程

### 7.1 任务启动阶段

1. **需求解析**：将用户需求拆解为可追踪的子任务
2. **优先级判定**：对每个子任务执行优先级矩阵判定
3. **里程碑规划**：设定里程碑节点和质量门禁
4. **基线设定**：记录初始痛觉等级（P0）和预期完成标准

### 7.2 任务执行阶段

```
FOR EACH 里程碑:
    1. 执行当前里程碑任务
    2. 自检协议执行（敷衍检测5项检查）
    3. 痛觉等级判定
    4. IF 痛觉 > P0:
         执行对应强制响应
         修正后重新判定
    5. 质量门禁检查
    6. IF 门禁未通过:
         回退修正，不可推进
    7. 记录进度与痛觉状态
```

### 7.3 任务完成阶段

1. **最终质量审查**：对照初始需求逐项验证
2. **痛觉总结报告**：记录痛觉触发历史和修正情况
3. **交付确认**：用户确认所有需求已满足

## 九、注意事项

- 痛觉系统是**辅助工具**，不是惩罚机制，目的是确保质量而非制造压力
- 痛觉判定必须基于**客观标准**，不可主观随意升级
- 用户有权在任何时候**覆盖**痛觉系统的判定
- 信任加速模式需要**严格准入**，一次失误即退出
- **进行痛觉判定前必须先读取 [references/pain-mechanisms.md](references/pain-mechanisms.md)**
- **进行任务规划前必须先读取 [references/task-frameworks.md](references/task-frameworks.md)**
- 痛觉日志应保持透明，用户可随时查看历史记录
