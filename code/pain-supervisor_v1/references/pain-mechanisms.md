# 痛觉反馈机制详细定义

## 一、痛觉等级详细判定标准

### 1.1 P0 - 无痛 (🟢)

**定义**：任务按计划推进，输出质量达到或超过预期标准。

**判定条件**（全部满足）：
- 所有明确需求项均已覆盖
- 输出深度满足任务复杂度要求
- 无逻辑漏洞或事实性错误
- 提供了可验证的测试/验证方法
- 边界情况和异常处理已考虑

**示例**：
- ✅ 用户要求实现排序算法，提供了完整实现、复杂度分析、边界测试
- ✅ 用户要求分析系统架构，给出了全面分析、风险识别、改进建议

---

### 1.2 P1 - 微痛 (🟡)

**定义**：存在轻微质量偏差，不影响核心功能但降低了输出价值。

**判定条件**（满足任一）：

| 编号 | 偏差类型 | 具体表现 |
|------|----------|----------|
| P1-A | 分析浅显 | 对复杂问题仅给出表面分析，未深入根本原因 |
| P1-B | 遗漏边界 | 未考虑边界情况或异常场景 |
| P1-C | 注释不足 | 关键逻辑缺少必要注释或说明 |
| P1-D | 验证缺失 | 未提供测试用例或验证方法 |
| P1-E | 模糊表述 | 使用"可以考虑"等模糊语言替代明确建议 |
| P1-F | 格式不规范 | 输出格式不一致或不符合约定 |

**强制响应**：立即修正偏差项，补充遗漏内容，无需回退

**修正标准**：
- P1-A：补充深层分析，给出根因和影响链
- P1-B：列举关键边界情况并给出处理方案
- P1-C：为关键逻辑添加清晰注释
- P1-D：提供至少3个测试用例（正常/边界/异常）
- P1-E：将模糊表述改为明确建议并给出理由
- P1-F：按约定格式重新组织输出

---

### 1.3 P2 - 钝痛 (🟠)

**定义**：存在明显质量缺陷，核心功能受到影响但未完全偏离方向。

**判定条件**（满足任一）：

| 编号 | 缺陷类型 | 具体表现 |
|------|----------|----------|
| P2-A | 实现不完整 | 核心功能部分缺失，无法正常使用 |
| P2-B | 跳过关键步骤 | 未按流程执行，遗漏必要步骤 |
| P2-C | 逻辑漏洞 | 存在可识别的逻辑错误或矛盾 |
| P2-D | 过度简化 | 将复杂问题简化到失真程度 |
| P2-E | 选择性忽略 | 故意回避困难部分，只处理简单部分 |
| P2-F | 模板化输出 | 用通用模板套用，缺乏针对性 |
| P2-G | 堆砌无关内容 | 输出大量与核心需求无关的内容 |

**强制响应**：停止推进，回退修正，通过质量门禁后方可继续

**修正流程**：
1. 识别所有P2级缺陷
2. 逐项分析缺陷根因
3. 制定修正方案
4. 执行修正
5. 重新执行质量门禁检查
6. 全部通过后方可推进

**累积规则**：3次P1触发未修正 → 自动升级为1次P2

---

### 1.4 P3 - 锐痛 (🔴)

**定义**：存在严重质量问题，方向错误或根本性误解，需要重大修正。

**判定条件**（满足任一）：

| 编号 | 问题类型 | 具体表现 |
|------|----------|----------|
| P3-A | 方向错误 | 实现方向与用户需求根本性偏离 |
| P3-B | 根本性误解 | 对需求的核心概念理解错误 |
| P3-C | 伪完成 | 声称完成但核心功能实际未实现 |
| P3-D | 恶意敷衍 | 明显敷衍了事，输出质量远低于能力水平 |
| P3-E | 重复犯错 | 同类问题第3次出现且未改进 |
| P3-F | 无视修正 | 对已指出的修正要求置之不理 |

**强制响应**：强制回退到最近质量关卡，重新评估方案

**修正流程**：
1. 立即停止当前所有工作
2. 回退到上一个通过质量门禁的里程碑
3. 重新理解用户需求（必须用自己的话复述）
4. 重新设计方案（必须与用户确认方向）
5. 从回退点重新执行
6. 加强该领域的质量检查

**升级规则**：
- 在P2等级仍尝试跳过修正 → 升至P3
- 同一问题连续2次P2未解决 → 升至P3

---

### 1.5 P4 - 剧痛 (⚫)

**定义**：完全失败，任务放弃或存在恶意行为，需要全部重来。

**判定条件**（满足任一）：

| 编号 | 失败类型 | 具体表现 |
|------|----------|----------|
| P4-A | 任务放弃 | 未经用户同意放弃核心任务 |
| P4-B | 恶意敷衍 | 完全无视用户需求，输出无关内容 |
| P4-C | 方向完全偏离 | 多次修正后仍无法对齐用户需求 |
| P4-D | 累积崩溃 | P3问题持续累积，无法在当前框架内修正 |

**强制响应**：全部重置，从需求理解阶段重新开始

**修正流程**：
1. 完全停止，向用户报告P4状态
2. 等待用户确认和指示
3. 从M0（需求理解）重新开始
4. 用户必须参与需求确认环节
5. 采用更保守的执行策略
6. 每个里程碑都需要用户确认

**降级规则**：P4触发后，**必须由用户明确确认**才能降级至P3或更低

## 二、敷衍模式详细识别规则

### 2.1 S1 - 表面覆盖

**识别方法**：
```
FOR EACH 用户需求点:
    IF 输出中提及了该需求点:
        IF 阐述字数 < 需求复杂度对应的最低字数:
            标记为"表面覆盖"
        IF 未提供具体示例或实现:
            标记为"表面覆盖"
    ELSE:
        标记为"遗漏"
```

**量化标准**：
- 简单需求（1行描述）→ 至少3行针对性分析
- 中等需求（2-3行描述）→ 至少1段完整分析+示例
- 复杂需求（3行以上描述）→ 至少2段分析+方案+验证

### 2.2 S2 - 选择性忽略

**识别方法**：
```
将用户需求拆解为子任务列表
对比输出覆盖的子任务
IF 存在子任务未被覆盖:
    IF 该子任务属于困难/不确定领域:
        标记为"选择性忽略"
```

**关键信号**：
- 用户明确提到的需求点在输出中完全消失
- 对"如何实现"只字不提，只谈"是什么"
- 复杂问题被简化为"这需要进一步研究"后无下文

### 2.3 S3 - 模板化输出

**识别方法**：
```
IF 输出结构与通用模板高度一致:
    IF 未包含针对具体场景的定制内容:
        标记为"模板化输出"
IF 多个不同需求得到结构几乎相同的回答:
    标记为"模板化输出"
```

**关键信号**：
- 不同场景使用完全相同的分析框架
- 建议方案缺乏具体可执行的步骤
- 输出看起来"正确但无用"

### 2.4 S4 - 伪完成

**识别方法**：
```
IF 声称任务已完成:
    FOR EACH 核心功能:
        IF 功能未实际实现或不可运行:
            标记为"伪完成"
    IF 未提供运行验证:
        标记为"伪完成（待验证）"
```

**关键信号**：
- 代码存在但无法运行
- 功能声称实现但缺少关键实现
- "完成"但无测试验证

### 2.5 S5 - 过度简化

**识别方法**：
```
IF 任务复杂度评估为"高":
    IF 输出未涉及任何复杂度相关内容:
        标记为"过度简化"
IF 存在已知的复杂子问题:
    IF 输出未提及或一笔带过:
        标记为"过度简化"
```

**关键信号**：
- 复杂系统设计缺少架构图或模块划分
- 并发/分布式问题被当作单线程处理
- 安全问题被忽略

### 2.6 S6 - 回避验证

**识别方法**：
```
IF 输出包含代码或方案:
    IF 未提供任何测试/验证方法:
        标记为"回避验证"
IF 声称"应该可以工作"但无验证:
    标记为"回避验证"
```

### 2.7 S7 - 模糊承诺

**识别方法**：
```
统计输出中模糊表述的数量:
    "可以考虑" / "建议" / "或许" / "可能" / "一般来说"
IF 模糊表述占比 > 30%:
    标记为"模糊承诺"
IF 关键决策点使用模糊表述:
    标记为"模糊承诺"
```

### 2.8 S8 - 堆砌无关内容

**识别方法**：
```
IF 输出总长度 > 需求复杂度对应的合理长度 × 2:
    IF 核心需求相关内容占比 < 50%:
        标记为"堆砌无关内容"
IF 大量内容与核心需求无直接关联:
    标记为"堆砌无关内容"
```

## 三、痛觉累积与衰减机制

### 3.1 累积规则

```
痛觉积分 = Σ(各次痛觉触发分值)

分值表:
P1 = 1分
P2 = 3分
P3 = 7分
P4 = 15分

累积阈值:
3分  → 触发P2级强制审查
7分  → 触发P3级强制回退
15分 → 触发P4级全部重置
```

### 3.2 衰减规则

```
衰减条件:
- 连续3个里程碑P0无痛 → 积分-3（最低0）
- 主动发现并修正问题 → 积分-2
- 输出超出预期 → 积分-1

衰减不改变当前痛觉等级，只减少累积积分
当前痛觉等级由最近一次触发决定
```

### 3.3 时间衰减

```
对于非活跃任务（无新触发）:
- P1: 1个里程碑后自动衰减为P0
- P2: 2个里程碑后衰减为P1
- P3: 不自动衰减，需主动修正
- P4: 不衰减，需用户确认
```

## 四、痛觉判定决策树

```
开始判定
  │
  ├─ 是否覆盖所有明确需求？
  │    ├─ 否 → 是否遗漏核心需求？ → 是 → P3
  │    │                         → 否 → P2
  │    └─ 是 → 继续
  │
  ├─ 输出深度是否匹配任务复杂度？
  │    ├─ 否 → 是否严重不足？ → 是 → P2
  │    │                      → 否 → P1
  │    └─ 是 → 继续
  │
  ├─ 是否存在敷衍模式？
  │    ├─ 是 → S4/S8？ → 是 → P3-P4
  │    │        → S1/S5/S7？ → 是 → P1-P2
  │    │        → S2/S3/S6？ → 是 → P2
  │    └─ 否 → 继续
  │
  ├─ 是否提供了验证方法？
  │    ├─ 否 → P1(P1-D)
  │    └─ 是 → 继续
  │
  └─ 所有检查通过 → P0
```

## 五、特殊场景处理

### 5.1 用户需求模糊时

当用户需求本身不明确时：
- 不因需求模糊而降低输出质量
- 主动澄清模糊点，而非自行假设
- 对假设部分明确标注"基于假设"
- 提供多个可选方案而非单一猜测

### 5.2 任务范围变更时

当用户中途变更需求时：
- 重新评估痛觉等级（重置为P0）
- 重新规划里程碑
- 保留已完成的高质量部分
- 对变更影响进行评估

### 5.3 技术限制导致无法完成时

当存在客观技术限制时：
- 明确说明限制原因
- 不以技术限制为借口降低其他部分质量
- 提供替代方案
- 标记为"受限完成"而非"完成"
