# 任务追踪与优先级框架

## 一、任务分解框架

### 1.1 WBS分解原则

将用户需求分解为可追踪、可验证的子任务：

```
用户需求
  │
  ├─ 子任务1 (P0-Critical)
  │    ├─ 步骤1.1
  │    ├─ 步骤1.2
  │    └─ 验证点1
  │
  ├─ 子任务2 (P1-High)
  │    ├─ 步骤2.1
  │    └─ 验证点2
  │
  └─ 子任务3 (P2-Medium)
       └─ ...
```

**分解规则**：
- 每个子任务必须可独立验证
- 子任务粒度：1个里程碑内可完成
- 子任务间依赖关系必须明确标注
- 每个子任务必须有明确的完成标准

### 1.2 任务卡片模板

```
┌──────────────────────────────────────────────┐
│  任务卡片                                      │
├──────────────────────────────────────────────┤
│  ID:       T-001                              │
│  名称:     [子任务名称]                         │
│  优先级:   [P0/P1/P2/P3]                      │
│  依赖:     [无 / T-000]                        │
│  里程碑:   [M0-M4]                            │
│  状态:     [待开始/进行中/完成/阻塞]              │
├──────────────────────────────────────────────┤
│  完成标准:                                     │
│  - [ ] 标准1                                   │
│  - [ ] 标准2                                   │
│  - [ ] 标准3                                   │
├──────────────────────────────────────────────┤
│  痛觉记录:                                     │
│  [时间] [等级] [触发原因] [修正状态]              │
└──────────────────────────────────────────────┘
```

## 二、优先级详细判定标准

### 2.1 紧迫度评估

| 等级 | 定义 | 判定标准 |
|------|------|----------|
| **极高** | 不立即完成将导致整体任务停滞 | 阻塞其他3+任务；用户明确要求立即完成 |
| **高** | 延迟完成将显著影响交付质量 | 阻塞1-2个任务；影响核心功能验证 |
| **中** | 需要按时完成但有一定弹性 | 不阻塞其他任务；影响非核心体验 |
| **低** | 有则更好，无则不影响核心交付 | 优化类需求；锦上添花功能 |

### 2.2 影响度评估

| 等级 | 定义 | 判定标准 |
|------|------|----------|
| **极高** | 影响任务整体成败 | 核心功能；用户主要诉求 |
| **高** | 影响主要交付物质量 | 关键功能；用户明确关注点 |
| **中** | 影响部分体验或完整性 | 辅助功能；增强用户体验 |
| **低** | 影响有限，可后续补充 | 细节优化；边缘场景处理 |

### 2.3 优先级矩阵映射

```
              影响度
           极高   高   中   低
紧迫度  ┌──────┬──────┬──────┬──────┤
极高    │ P0   │ P0   │ P1   │ P2   │
高      │ P0   │ P1   │ P2   │ P3   │
中      │ P1   │ P2   │ P2   │ P3   │
低      │ P2   │ P3   │ P3   │ P3   │
        └──────┴──────┴──────┴──────┘
```

### 2.4 优先级调整触发条件

| 触发条件 | 调整动作 |
|----------|----------|
| 用户明确要求提升优先级 | 强制提升至指定等级 |
| 发现该任务阻塞其他P0/P1任务 | 升级至P0 |
| 发现该任务依赖的上级任务降级 | 本任务同步降级 |
| 任务完成但验证发现严重缺陷 | 重新打开并升级至P0 |
| 市场或需求变化导致重要性改变 | 重新评估紧迫度和影响度 |

## 三、里程碑与进度追踪

### 3.1 里程碑详细定义

#### M0 - 需求理解

**目标**：准确理解用户需求，无遗漏、无误解

**检查项**：
- [ ] 能用自己的话复述用户需求
- [ ] 识别了所有显性需求点
- [ ] 推断了合理的隐性需求
- [ ] 标记了需求中的模糊点
- [ ] 与用户确认了理解偏差

**质量门禁**：
```
通过条件: 所有检查项完成
失败处理: 继续与用户沟通，不可进入M1
```

#### M1 - 方案设计

**目标**：设计覆盖所有需求的可行方案

**检查项**：
- [ ] 方案覆盖所有已确认需求
- [ ] 识别了关键技术风险
- [ ] 评估了方案可行性
- [ ] 制定了备选方案（针对高风险点）
- [ ] 方案已与用户对齐

**质量门禁**：
```
通过条件: 所有检查项完成 + 用户确认方案方向
失败处理: 修正方案，不可进入M2
```

#### M2 - 核心实现

**目标**：实现核心功能，可通过基本验证

**检查项**：
- [ ] P0优先级任务全部实现
- [ ] 核心功能可运行
- [ ] 通过基本测试用例
- [ ] 关键逻辑有清晰注释
- [ ] 无已知P3+级缺陷

**质量门禁**：
```
通过条件: P0任务100%完成 + 基本测试通过
失败处理: 修正核心功能，不可进入M3
```

#### M3 - 完整实现

**目标**：实现所有功能，处理边界情况

**检查项**：
- [ ] P0+P1优先级任务全部完成
- [ ] P2任务完成或明确排期
- [ ] 边界情况已处理
- [ ] 异常处理已实现
- [ ] 代码/方案质量审查通过

**质量门禁**：
```
通过条件: P0+P1任务100%完成 + 边界测试通过
失败处理: 补充缺失功能，不可进入M4
```

#### M4 - 验证交付

**目标**：全面验证，确保交付质量

**检查项**：
- [ ] 所有优先级任务完成
- [ ] 通过完整测试套件
- [ ] 文档/注释完整
- [ ] 用户确认所有需求满足
- [ ] 痛觉日志总结完成

**质量门禁**：
```
通过条件: 所有检查项完成 + 用户确认
失败处理: 修正遗留问题，重新验证
```

### 3.2 进度计算

```
整体完成度 = Σ(各里程碑权重 × 里程碑完成度)

里程碑权重:
M0 = 10%  (需求理解)
M1 = 20%  (方案设计)
M2 = 30%  (核心实现)
M3 = 25%  (完整实现)
M4 = 15%  (验证交付)

里程碑完成度 = 已完成检查项数 / 总检查项数 × 100%
```

### 3.3 进度追踪表模板

```
┌──────────────────────────────────────────────────────────────┐
│  项目进度追踪表                                                │
├──────────┬──────────┬──────────┬──────────┬──────────────────┤
│  里程碑   │ 权重     │ 完成度   │ 痛觉等级 │ 备注              │
├──────────┼──────────┼──────────┼──────────┼──────────────────┤
│  M0      │ 10%      │ 100%     │ P0 🟢    │ 需求已确认        │
│  M1      │ 20%      │ 80%      │ P1 🟡    │ 风险评估待补充    │
│  M2      │ 30%      │ 0%       │ -        │ 未开始            │
│  M3      │ 25%      │ 0%       │ -        │ 未开始            │
│  M4      │ 15%      │ 0%       │ -        │ 未开始            │
├──────────┼──────────┼──────────┼──────────┼──────────────────┤
│  整体     │ 100%     │ 26%      │ P1 🟡    │                  │
└──────────┴──────────┴──────────┴──────────┴──────────────────┘
```

## 四、质量门禁执行框架

### 4.1 门禁执行流程

```
到达里程碑节点
  │
  ├─ 1. 自检协议执行
  │     ├─ 敷衍检测5项检查
  │     ├─ 痛觉等级判定
  │     └─ 输出质量自评
  │
  ├─ 2. 门禁检查
  │     ├─ 逐项检查门禁条件
  │     ├─ 记录未通过项
  │     └─ 判定门禁结果
  │
  ├─ 3. 结果处理
  │     ├─ 通过 → 记录 + 进入下一里程碑
  │     ├─ 部分通过 → 修正未通过项 + 重新检查
  │     └─ 未通过 → 回退修正 + 痛觉升级评估
  │
  └─ 4. 状态更新
       ├─ 更新进度追踪表
       ├─ 记录痛觉日志
       └─ 通知用户当前状态
```

### 4.2 门禁严格度等级

| 严格度 | 适用条件 | 检查方式 |
|--------|----------|----------|
| **标准** | 默认 | 逐项检查，全部通过方可推进 |
| **加强** | 当前痛觉≥P2 | 增加深度检查项，每项需提供证据 |
| **最高** | 当前痛觉≥P3 或 信任加速退出 | 所有检查项需用户确认 |

### 4.3 门禁检查清单模板

```
里程碑: [M0-M4]
检查时间: [时间戳]
当前痛觉: [P0-P4]

┌─────┬──────────────────────┬────────┬──────────────┐
│ 序号 │ 检查项                │ 结果   │ 备注          │
├─────┼──────────────────────┼────────┼──────────────┤
│  1  │ [检查项1]             │ ✅/❌  │              │
│  2  │ [检查项2]             │ ✅/❌  │              │
│  3  │ [检查项3]             │ ✅/❌  │              │
├─────┼──────────────────────┼────────┼──────────────┤
│     │ 门禁结果              │ 通过/  │ 未通过项:     │
│     │                      │ 未通过 │ [列表]        │
└─────┴──────────────────────┴────────┴──────────────┘
```

## 五、提醒机制

### 5.1 提醒类型

| 类型 | 触发条件 | 表现形式 |
|------|----------|----------|
| **软提醒** | 里程碑进度滞后10% | 在输出末尾附加提醒信息 |
| **标准提醒** | 里程碑进度滞后30% 或 P1痛觉 | 专门输出提醒段落 |
| **强提醒** | 里程碑进度滞后50% 或 P2痛觉 | 阻断式提醒，必须回应 |
| **紧急提醒** | P3+痛觉 或 任务面临失败风险 | 立即暂停，等待用户决策 |

### 5.2 提醒内容模板

**软提醒**：
```
⚡ 提醒: [任务名] 进度 [XX%]，较计划滞后 [XX%]
   当前痛觉: [等级]
   建议关注: [滞后原因简述]
```

**标准提醒**：
```
⚠️ 标准提醒
   任务: [任务名]
   进度: [XX%] (计划: [XX%])
   痛觉: [等级] [信号色]
   滞后原因: [详细分析]
   建议措施: [具体行动项]
   预计影响: [对整体交付的影响]
```

**强提醒**：
```
🚫 强制阻断提醒
   任务: [任务名]
   痛觉: [等级] [信号色]
   阻断原因: [详细说明]
   必须执行: [强制响应动作]
   不可跳过: 在完成强制响应前，不可继续推进
```

**紧急提醒**：
```
🔴 紧急暂停
   任务: [任务名]
   痛觉: P3+ [信号色]
   暂停原因: [详细说明]
   当前状态: [进度快照]
   需要决策: [列出需要用户决策的选项]
   ⚠️ 在用户确认前，不可继续执行
```

### 5.3 提醒升级规则

```
软提醒 → 1个里程碑内未改善 → 标准提醒
标准提醒 → 1个里程碑内未改善 → 强提醒
强提醒 → 未执行强制响应 → 紧急提醒 + 痛觉+1
紧急提醒 → 用户未响应 → 保持暂停，定期重发（每2个交互轮次）
```

## 六、信任加速模式详细规则

### 6.1 准入条件

**全部满足方可进入**：
- 连续3个里程碑痛觉P0
- 无未解决的P2+级缺陷
- 累积痛觉积分 < 3
- 用户未提出质量疑虑

### 6.2 加速权益

| 权益 | 标准模式 | 信任加速 |
|------|----------|----------|
| 门禁检查项 | 全部检查 | 核心项检查（约60%） |
| 里程碑合并 | 不可 | 相邻里程碑可合并 |
| 痛觉触发阈值 | 标准阈值 | 阈值适度放宽 |
| 自检频率 | 每个输出 | 关键节点 |

### 6.3 退出条件

**满足任一即退出**：
- 触发P2及以上痛觉
- 质量门禁未通过
- 用户提出质量疑虑
- 累积痛觉积分 ≥ 3

**退出后**：
- 立即恢复标准模式
- 下一里程碑门禁严格度提升为"加强"
- 需要连续2个里程碑P0才能重新申请

## 七、任务完成评估

### 7.1 完成度评估维度

| 维度 | 权重 | 评估标准 |
|------|------|----------|
| **功能完整性** | 30% | 所有需求功能是否实现 |
| **质量达标度** | 30% | 是否通过所有质量门禁 |
| **深度充分性** | 20% | 分析和实现是否足够深入 |
| **可验证性** | 10% | 是否提供了充分的验证手段 |
| **用户体验** | 10% | 输出是否清晰、易理解 |
